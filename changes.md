# Изменения во Flutter-приложении — план тестирования после мерджа

Документ описывает изменения, сделанные в коммите `Parking attempts added`
(репозиторий `Filanteyee/Smart-Res-Flutter`, ветка `main`), и пошаговый план
проверки, что всё работает после мерджа с веткой тиммейта.

---

## 1. Что нового во Flutter-приложении

### 1.1. `lib/pages/parking_page.dart` — переделана страница «Паркинг»

**Удалено:**
- Раздел «Гостевые места» (сетка свободных/занятых мест с возможностью забронировать).
- Раздел «Мои бронирования» (список броней резидента с кнопкой отмены).
- Окно бронирования места (`_BookingSheet` с выбором даты/времени «С» и «По»).
- Сообщение администратору о неправомерно занятом гостевом месте.

**Добавлено — система «3 попыток» на получение пропуска:**
- Счётчик «Отклонено заявок: X / 3» (оранжевый, после 3-х — красный).
- После 3 отказов вместо «Подать заявку» появляется кнопка
  «**Обратиться в поддержку**», которая открывает Service Requests
  с предзаполненным описанием «Апелляция по паркингу».
- Если апелляция тоже отклонена — красная плашка:
  *«Апелляция отклонена. Для решения вопроса обратитесь лично
  в управляющую компанию.»*

**Добавлено — просмотр документа пропуска:**
- Кнопка «Открыть документ» на карточке одобренного пропуска.
- Загрузка файла через `ApiClient.downloadBytes` (с подменой
  `localhost`/`127.0.0.1`/`10.0.2.2` на реальный хост).
- Картинки открываются в `InteractiveViewer` (с зумом).
- PDF открываются в новом `_DocViewerPage` через `flutter_pdfview`.
- Остальные форматы открываются системным приложением (`open_file`).

**Прочее:**
- В диалоге подачи заявки нельзя выбрать место с `assigned_user_id != null`.
- Новые зависимости: `dio`, `flutter_pdfview`, `open_file`, `path_provider`.

### 1.2. `lib/pages/service_requests_page.dart` — поддержка апелляций

- Новый параметр `prefilledTitle` в конструкторе `ServiceRequestsPage`. При
  заходе с этим параметром автоматически открывается лист создания заявки
  с предзаполненным описанием.
- Новый метод `_resolveAppeal(id, approved)` —
  `POST /admin/service-requests/{id}/resolve-appeal` с телом
  `{"approved": bool}`. Показывает снэкбар «Пропуск одобрен» или
  «Апелляция отклонена».
- В карточке заявки `_RequestCard` для админа появляется блок
  «Апелляция по паркингу» (виден, если в описании есть
  «Апелляция по паркингу» и статус не `done`/`rejected`):
  - Зелёная кнопка «**Одобрить пропуск**».
  - Красная outline-кнопка «**Отклонить**».

### 1.3. `lib/pages/admin_verification_page.dart` — диалог отклонения пропуска

- Раньше: поле комментария было необязательным.
- Теперь:
  - Сверху диалога — кнопка «**Просмотреть документ**» (если документ есть);
    открывает прикреплённый файл прямо из диалога.
  - Поле комментария обязательное (3 строки, hint «Укажите причину»).
  - Кнопка «Отклонить» disabled, пока поле пустое (через `StatefulBuilder` +
    `setDialogState`).

### 1.4. `lib/pages/dashboard_page.dart` — доступ к паркингу

- Из `/profile` теперь читается новое поле `parking_permit_status`.
- Геттер `_hasParkingAccess` = `verificationStatus == 'approved'`
  **или** `parkingPermitStatus == 'approved'`.
- `_ParkingStatusCard` показывает режим «резидент», если выполнено хотя бы
  одно из этих условий (раньше — только верификация).
- При логауте `_parkingPermitStatus` сбрасывается в `''`.

### 1.5. `lib/widgets/create_request_sheet.dart` — предзаполнение описания

- Новый параметр `prefilledDescription` в конструкторе.
- В `initState` поле описания сразу заполняется этим текстом.
- Используется при «Обратиться в поддержку» → «Апелляция по паркингу».

### 1.6. `lib/pages/guests_page.dart` — статус `arrived`

- Новый статус гостевого пропуска `arrived` → отображается как
  «На территории».
- `_myActivePassSpotIds` теперь включает пропуска со статусом и `active`,
  и `arrived` — пока гость на территории, его место остаётся занятым.

## 2. Что должно быть на бэкенде

Без этих эндпоинтов и полей фронт сломается молча или с сетевыми ошибками.

| Что нужно | Где используется на фронте |
|---|---|
| `POST /admin/service-requests/:id/resolve-appeal` принимает `{"approved": bool}` | `service_requests_page.dart` → `_resolveAppeal` |
| `GET /profile` возвращает поле `parking_permit_status` (значения `approved`, `pending`, `rejected`, …) | `dashboard_page.dart` |
| У `ParkingSpot` есть `assigned_user_id` (null если место можно выбирать) | `parking_page.dart` → диалог подачи заявки |
| Документ пропуска отдаётся по URL, доступному через `ApiClient.downloadBytes` | `parking_page.dart` → «Открыть документ» |
| Гостевые пропуска поддерживают статус `arrived` | `guests_page.dart` |
| `GET /service-requests` возвращает заявки резидента (для определения статуса апелляции по `description = 'Апелляция по паркингу'` и `status = 'rejected'`) | `parking_page.dart` → `_appealRejected` |

## 3. Шаги после мерджа

```bash
flutter pub get
flutter analyze
flutter run
```

Если `flutter analyze` показывает ошибки — мердж что-то сломал в типах
или импортах. Чинить до запуска.

**Новые пакеты, которые должны подтянуться:** `dio`, `flutter_pdfview`,
`open_file`, `path_provider`. Если их нет в `pubspec.yaml` — мердж
`pubspec.yaml` пошёл не так.

## 4. Чек-лист тестирования

### 4.1. Резидент — заявка на пропуск (3 попытки)

- [ ] Залогиниться как резидент с одобренной верификацией.
- [ ] Открыть «Паркинг» → секции «Гостевые места» и «Мои бронирования» **отсутствуют**.
- [ ] Видна секция «Пропуск», нет активного пропуска → кнопка «Подать заявку».
- [ ] Подать заявку, админом отклонить → у резидента видно
      «**Отклонено заявок: 1 / 3**» (оранжевое).
- [ ] Подать ещё две заявки, обе отклонить → счётчик «**3 / 3**» (красное),
      вместо «Подать заявку» появилась оранжевая кнопка
      «**Обратиться в поддержку**».
- [ ] Нажать «Обратиться в поддержку» → открывается экран Service Requests
      с автоматически открытым листом создания заявки, в поле описания
      уже стоит «**Апелляция по паркингу**».
- [ ] Отправить заявку → она появляется в списке Service Requests.

### 4.2. Админ — апелляция

- [ ] Залогиниться админом, открыть Service Requests.
- [ ] Найти заявку с описанием «Апелляция по паркингу» → в карточке виден блок:
  - Зелёная кнопка «**Одобрить пропуск**».
  - Красная кнопка «**Отклонить**».
- [ ] Нажать «Одобрить пропуск» → снэкбар «Пропуск одобрен», у резидента
      пропуск становится approved.
- [ ] (На другой заявке) Нажать «Отклонить» → у резидента в «Паркинге»
      красная плашка «**Апелляция отклонена. Для решения вопроса обратитесь
      лично в управляющую компанию.**», кнопок подачи больше нет.

### 4.3. Админ — обработка пропуска

- [ ] Открыть админскую панель → вкладка «Пропуска».
- [ ] У заявки с прикреплённым документом нажать «Отклонить».
- [ ] В открывшемся диалоге:
  - [ ] Сверху есть кнопка «**Просмотреть документ**», она работает
        (открывает файл).
  - [ ] Поле комментария пустое, hint «Укажите причину».
  - [ ] Кнопка «**Отклонить**» **disabled** пока поле пустое.
  - [ ] Ввести любой текст → кнопка стала активной → отклонение проходит.

### 4.4. Резидент — просмотр документа

- [ ] У резидента с одобренным пропуском и прикреплённым документом.
- [ ] Открыть «Паркинг» → на карточке пропуска видна кнопка
      «**Открыть документ**».
- [ ] Нажать:
  - [ ] Если файл — картинка: открывается экран с InteractiveViewer (можно зумить).
  - [ ] Если PDF: открывается встроенный PDF-вьюер (`flutter_pdfview`).
  - [ ] Если другой формат: открывается системным приложением (`open_file`).

### 4.5. Дашборд — доступ к паркингу

- [ ] Резидент с **не одобренной верификацией**, но одобренным
      `parking_permit_status=approved` → карточка паркинга на дашборде
      в режиме «резидент» (не «не верифицирован»).
- [ ] Логаут → карточка возвращается в режим «гость».

### 4.6. Гостевой пропуск

- [ ] Создать гостевой пропуск с привязкой к месту.
- [ ] Перевести его в статус `arrived` (через бэк или соответствующий флоу).
- [ ] На странице «Гости» статус отображается как «**На территории**».
- [ ] Это место по-прежнему числится занятым (не освобождается, пока гость
      на территории).

### 4.7. Запрет выбора занятого места

- [ ] При подаче заявки на пропуск в диалоге выбора места: место с
      `assigned_user_id != null` показано красным и **не нажимается**.

- `flutter analyze` показывает ошибки → проблема в коде (конфликт мерджа).
  Показать Егору, что именно.
- Приложение собирается, но фича не работает → скорее всего, бэкенд не
  поддерживает что-то из раздела «Что должно быть на бэкенде». Проверь
  сетевые запросы в логах / DevTools Network.
- Что-то ведёт себя иначе, чем в чек-листе → это регрессия, нужно разбираться.

## 6. Изменения на бэкенде — коммит «Added parking attempts»

Разница между коммитами `384458a` (changes) и `36a830b` (Added parking attempts).

### 6.1. Что добавлено

**`service_requests.go` + `main.go` — апелляция по паркингу**
- Новый хэндлер `ResolveAppeal`: admin одобряет или отклоняет апелляцию.
- При одобрении: автоматически находит последний `rejected` пропуск резидента,
  меняет статус → `approved`, обновляет `profiles.parking_permit_status = 'approved'`,
  закрывает заявку (`done`).
- Роут: `POST /api/v1/admin/service-requests/:id/resolve-appeal` с телом `{"approved": bool}`.

**`parking_permit.go` — лимит попыток и обязательный комментарий**
- Ограничение в **3 попытки**: если `rejectedCount >= 3` → 403 `max_rejections_reached`.
- При отклонении пропуска поле `admin_comment` обязательно → 400 если пустое.
- Уведомления об «Unknown Vehicle» теперь всегда сохраняются в БД.

**`barrier_v2.go` — двойное сканирование QR**
- Гость с `parking_spot_id`: первое сканирование QR = `arrived` + `QR_SCAN_TERRITORY`,
  второе сканирование (у паркинг-ворот) = `used` + `QR_SCAN_PARKING`.
- По номерному знаку: гость с зарезервированным местом → `arrived`, без места → `used`.
- Уведомления всегда пишутся в БД независимо от наличия FCM-токена.

**`types.go` + `mqtt/client.go` — статус пропуска в профиле**
- `ParkingPermitStatus` добавлен в `profileRow` → поле `parking_permit_status` в ответе `GET /me`.
- MQTT: проверка статуса гостя включает `'arrived'`; проверка занятости парковки
  учитывает события `QR_SCAN_PARKING`.

### 6.2. Как тестировать бэкенд после мержа

**Запуск:**
```
go run cmd/server/main.go
```
В логах не должно быть ошибок при старте (миграции, MQTT-подключение).

**Чек-лист:**

- [ ] `GET /api/v1/me` → ответ содержит поле `parking_permit_status`.
- [ ] Подать заявку на пропуск, отклонить 3 раза → на 4-й попытке ответ
      403 с `{"error":"max_rejections_reached"}`.
- [ ] Admin отклоняет пропуск без комментария → ответ 400.
- [ ] Admin одобряет апелляцию через `POST /admin/service-requests/:id/resolve-appeal`
      с `{"approved":true}` → у резидента `parking_permit_status` становится `approved`.
- [ ] Гостевой пропуск с `parking_spot_id`: первое QR-сканирование → статус `arrived`;
      второе → статус `used`.

**Если что-то не работает после мержа** — проверить конфликты в:
`barrier_v2.go`, `parking_permit.go`, `service_requests.go`.
